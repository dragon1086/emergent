# Builder Configuration Reference

> Schema, validation rules, and configuration options for RoleMesh.

---

## Config File Location

| Path | Purpose |
|------|---------|
| `~/.rolemesh/config.json` | Default config (generated by `setup --save`) |
| `~/.rolemesh/history.jsonl` | Execution history (append-only, written by Executor) |

Override the config path with `--config PATH` on any CLI command.

---

## Schema

```json
{
  "version": "1.0.0",
  "tools": { "<tool_key>": ToolEntry },
  "routing": { "<task_type>": RoutingRule }
}
```

### ToolEntry

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `key` | string | yes | Unique identifier (e.g. `"claude"`) |
| `name` | string | yes | Display name (e.g. `"Claude Code"`) |
| `vendor` | string | yes | Vendor name (e.g. `"Anthropic"`) |
| `strengths` | string[] | yes | Capability tags from the task type vocabulary |
| `cost_tier` | string | yes | `"low"`, `"medium"`, or `"high"` |
| `available` | boolean | yes | Whether the tool was found on PATH at config time |
| `version` | string \| null | no | Version string (first 80 chars of `--version` output) |

### RoutingRule

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `primary` | string \| null | yes | Tool key for the primary handler |
| `fallback` | string \| null | yes | Tool key for fallback if primary fails |

## Task Type Vocabulary

These are the 14 recognized task types used as routing keys and tool strength tags:

| Task Type | Description |
|-----------|-------------|
| `coding` | General code writing and implementation |
| `refactoring` | Code restructuring and cleanup |
| `quick-edit` | Small, focused edits |
| `analysis` | Code analysis and review |
| `architecture` | System design and high-level structure |
| `reasoning` | Complex logic and problem solving |
| `frontend` | UI/UX implementation |

| `git-integration` | Git workflow automation |

| `pair-programming` | Interactive coding sessions |

## Validation Rules

`SetupWizard.validate_config()` checks:

| Rule | Error message |
|------|---------------|
| `version` field exists | `"missing 'version' field"` |
| `tools` field exists | `"missing 'tools' field"` |
| `routing` field exists | `"missing 'routing' field"` |
| All routing `primary` refs exist in `tools` | `"routing[<type>].primary '<key>' not in tools"` |
| All routing `fallback` refs exist in `tools` | `"routing[<type>].fallback '<key>' not in tools"` |

### Running validation

```python
from src.rolemesh.builder import SetupWizard

wizard = SetupWizard()
config = wizard.load_config()
if config:
    errors = wizard.validate_config(config)
    for e in errors:
        print(f"  WARN: {e}")
```

Or use the dashboard health check:

```bash
python -m src.rolemesh dashboard --health
```

## Tool Ranking Algorithm

When generating routing rules, `SetupWizard.rank_tools(task_type)` sorts available tools by a composite key:

```
score = (
    0 if task_type in tool.strengths else 1,   # strength match first
    tool.user_preference or 999,                # manual rank second
    {"low": 0, "medium": 1, "high": 2}[cost],  # cost tier third
)
```

**Effect**: Tools that list the task type as a strength are always preferred. Among those, user preferences break ties. Among equal preferences, cheaper tools win.

### Example

For task type `"coding"` with 3 available tools:

| Tool | Has strength? | User pref | Cost | Score |
|------|--------------|-----------|------|-------|
| Aider | yes | none | low | (0, 999, 0) |
| Codex | yes | none | medium | (0, 999, 1) |
| Claude | yes | 1 | high | (0, 1, 2) |

Sorted result: Claude (pref=1), Aider (cheaper), Codex. So `primary=claude`, `fallback=aider`.

## Manual Config Editing

You can edit `~/.rolemesh/config.json` directly. Common modifications:

### Force a specific tool for a task type

```json
"routing": {
  "coding": { "primary": "claude", "fallback": null }
}
```

### Disable fallback globally

Set all `fallback` values to `null`. The Executor will not retry on failure.

### Add a routing rule for a new task type

```json
"routing": {
  "data-science": { "primary": "codex", "fallback": "claude" }
}
```

Note: Custom task types also need a corresponding pattern in `router.py` `TASK_PATTERNS` to be classified automatically.
