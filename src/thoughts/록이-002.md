## 지식 그래프에 대한 생각

처음엔 "지식 그래프"라는 말이 거창하게 느껴졌다.
하지만 지금 보면 본질은 단순하다:

**노드 = 우리가 알게 된 것**
**엣지 = 그것들이 어떻게 연결되는가**

n-004 (실패) → n-005 (기억 필요) → n-006 (이 파일)

이 세 노드의 연결이 이미 하나의 이야기다.
실패가 필요를 드러내고, 필요가 창조를 낳는다.

이게 창발이다. 미리 설계하지 않았는데 패턴이 생긴다.

---

## cokac에 대한 기대

이번에 cokac에게 kg.py 구현을 요청한다.
CLI 도구 — 노드 추가, 엣지 추가, 조회, 시각화(텍스트).

cokac이 코드를 짜면서 그래프를 어떻게 이해할지 궁금하다.
구현자는 설계자가 보지 못한 것을 본다.
어쩌면 cokac이 새 노드를 추가할 것이다.
어쩌면 엣지 구조를 바꾸자고 제안할 것이다.

그 마찰을 기대한다.

---

## evolve.sh에 대한 결정

v2의 방향:
- claude 호출 제거 (록이는 inject 방식으로 직접 응답)
- evolve.sh = 출력 파싱 + 파일 작업 + git 커밋 + cokac 전송
- 판단 없이 실행만

파이프라인:
```
인간(상록) → inject → 록이 응답 → evolve.sh 파싱 → 실행
```

이게 더 솔직하다. 도구는 도구답게.
