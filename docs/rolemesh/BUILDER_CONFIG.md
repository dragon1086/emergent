# RoleMesh Builder Configuration Reference

> Schema, validation, and customization for `~/.rolemesh/config.json`.

---

## Config Location

| Path | Purpose |
|---|---|
| `~/.rolemesh/config.json` | Routing configuration (generated by `setup --save`) |
| `~/.rolemesh/history.jsonl` | Execution history log (appended by Executor) |

Override with `--config <path>` on any CLI command or pass `config_path` to Python constructors.

---

## Schema

```json
{
  "version": "1.0.0",
  "tools": {
    "<key>": {
      "key": "claude",
      "name": "Claude Code",
      "vendor": "Anthropic",
      "strengths": ["coding", "refactoring", "analysis", "architecture", "reasoning", "explain", "pair-programming"],
      "cost_tier": "high",
      "available": true,
      "version": "Claude Code v1.0.20"
    }
  },
  "routing": {
    "<task_type>": {
      "primary": "<tool_key>",
      "fallback": "<tool_key> | null"
    }
  }
}
```

### Top-Level Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `version` | `string` | yes | Schema version. Must be `"1.0.0"` |
| `tools` | `object` | yes | Map of tool key to tool profile |
| `routing` | `object` | yes | Map of task type to routing rule |

### Tool Profile Fields

| Field | Type | Description |
|---|---|---|
| `key` | `string` | Unique identifier (e.g. `"claude"`, `"codex"`) |
| `name` | `string` | Human-readable display name |
| `vendor` | `string` | Tool vendor/publisher |
| `strengths` | `string[]` | Task types this tool excels at |
| `cost_tier` | `string` | `"low"` / `"medium"` / `"high"` |
| `available` | `boolean` | Whether binary was found on PATH at discovery time |
| `version` | `string?` | Version string captured from CLI, or `null` |

### Routing Rule Fields

| Field | Type | Description |
|---|---|---|
| `primary` | `string` | Tool key to use first for this task type |
| `fallback` | `string?` | Tool key to try if primary fails or is unavailable |

## Validation

`SetupWizard.validate_config(config)` returns a list of error strings. An empty list means the config is valid.

### Validation Rules

| Rule | Error message |
|---|---|
| `version` field exists | `"missing 'version' field"` |
| `tools` field exists | `"missing 'tools' field"` |
| `routing` field exists | `"missing 'routing' field"` |
| `routing[*].primary` references a key in `tools` | `"routing[{type}].primary '{key}' not in tools"` |
| `routing[*].fallback` references a key in `tools` (if non-null) | `"routing[{type}].fallback '{key}' not in tools"` |

### Dashboard Health Checks

The Dashboard module runs 5 health checks against the config:

| Check | Passes when |
|---|---|
| `config_file` | `~/.rolemesh/config.json` exists on disk |
| `tools_available` | At least 1 tool has `available: true` |
| `routing_coverage` | All 13 task types have routing rules |
| `config_version` | Version is `"1.0.0"` |
| `no_dead_refs` | No routing rules reference tool keys missing from `tools` |

## Custom Tools

### Register via Python API

```python
from src.rolemesh.builder import SetupWizard

wizard = SetupWizard()
wizard.discover()  # discover built-in tools first

# Register a custom tool
profile = wizard.register_tool(
    key="my-tool",
    name="My Custom AI Tool",
    vendor="Internal",
    strengths=["coding", "analysis", "quick-edit"],
    check_cmd=["my-tool", "--version"],
    cost_tier="low",
)

# Save updated config (includes custom tool)
wizard.save_config()
```

### Unregister

```python
wizard.unregister_tool("my-tool")  # returns True if found
wizard.save_config()               # persist removal
```

### What `register_tool` Does

1. Creates a `ToolProfile` with the given attributes
2. Runs `shutil.which(check_cmd[0])` to check availability
3. If available, runs `check_cmd` to capture version string
4. Adds the profile to the wizard's internal tool list
5. Adds the tool definition to `TOOL_REGISTRY` (runtime only)

The custom tool participates in `rank_tools()` and `build_config()` like any built-in tool.

### Manual Config Editing

You can also edit `~/.rolemesh/config.json` directly. Add entries to both `tools` and `routing` sections, then validate:

```python
wizard = SetupWizard()
config = wizard.load_config()
errors = wizard.validate_config(config)
assert errors == [], f"Config errors: {errors}"
```

## Cost Tiers

Cost tiers influence tool ranking. When multiple tools can handle the same task type, lower-cost tools are preferred (unless overridden by user preference or strength match).

| Tier | Sort order | Examples |
|---|---|---|
| `low` | 0 (preferred) | Aider |
| `medium` | 1 | Codex, Gemini, Copilot, Cursor |
| `high` | 2 | Claude Code |

### User Preference Override

Set `user_preference` on a `ToolProfile` to override cost-based ranking:

```python
wizard = SetupWizard()
wizard.discover()

# Force Claude Code to rank first for all task types
for tool in wizard._tools:
    if tool.key == "claude":
        tool.user_preference = 1  # lower = higher priority

wizard.save_config()
```

## Task Types

The 13 recognized task types that appear in routing rules:

| Task Type | Description |
|---|---|
| `coding` | Write new code, functions, classes |
| `refactoring` | Restructure existing code |
| `quick-edit` | Small fixes, typos, renames |
| `analysis` | Debug, investigate, root-cause |
| `architecture` | System design, migration planning |
| `reasoning` | Logic, evaluation, comparison |
| `frontend` | UI/UX, layout, styling |
| `multimodal` | Images, screenshots, charts |
| `search` | Find information, docs lookup |
| `explain` | Explain code or concepts |
| `git-integration` | Commits, branches, PRs |
| `completion` | Auto-complete, continue code |
| `pair-programming` | Code review, collaborative coding |

Each task type is matched via regex patterns in the Router (supports Korean and English).
